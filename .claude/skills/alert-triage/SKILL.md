# 告警分级技能

## When to Use

- 同一时间收到 3 条以上告警，需要分出优先级
- 告警涉及多个服务（payment-service、order-service、user-service），需要判断影响面
- 需要输出一份分级的告警处理清单

## When NOT to Use

- 只有一条告警时，直接分析那条告警，不需要分级
- 用户只是问某个告警的含义，直接解释

## 分级规则

### P0 - 紧急（需立即处理）
- 服务完全不可用（ServiceUnavailable）
- 数据库挂了
- 用户大面积无法访问

### P1 - 严重（30 分钟内处理）
- CPU 使用率 > 90%
- 内存使用率 > 90%，伴随 OOM
- P99 延迟 > 5s 且持续时间 > 15 分钟

### P2 - 警告（1 小时内处理）
- CPU/内存使用率 80-90%
- 响应时间 3-5s
- 慢查询数量上升

### P3 - 关注（下一个工作日）
- 低风险告警
- 属于已知问题且已有 workaround

## 调用流程

1. 调用 `query_prometheus_alerts` 获取当前所有活跃告警
2. 对每条告警判断级别、受影响服务和持续时间
3. 调用 `search_knowledge_base` 查询历史上的同类告警处理经验
4. 检查告警间关联关系（A 告警是否导致 B 告警？）
5. 按优先级输出处理清单

## 输出格式

```
## 告警分级结果

### 告警概览
活跃告警 X 条 | P0: 0 | P1: 2 | P2: 1 | P3: 1

### P0/P1 告警详情

| 级别 | 告警名称 | 服务 | 持续 | 影响评估 |
|-----|---------|------|------|---------|
| P1  | ...     | ...  | ...  | ...     |

### 建议处理顺序
1. [P1] xxx — 理由: xxx
2. [P1] yyy — 理由: yyy
3. [P2] zzz — 理由: zzz

### 关联分析
- 告警 A 和 B 高度相关，B 可能是 A 的衍生告警
- 先处理 A，B 可能随之消失
```
